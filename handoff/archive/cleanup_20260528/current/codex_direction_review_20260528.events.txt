# Authorized Bug Bounty Automation Platform Charter

## Mission

Build a user-owned, authorized bug bounty automation platform that continuously turns fresh vulnerability intelligence and target surface changes into operator-ready bounty decisions.

The platform should not be a general cybersecurity notebook. Its north star is a repeatable pipeline:

```text
fresh intel / target change -> detector or bundle lane -> candidate -> evidence packet -> operator decision -> report submission
```

## North Star Metrics

Primary outcomes:

- Operator inbox contains high-quality, bounded decisions instead of raw research.
- Each active lane has an EXECUTE / PARK / KILL state.
- Report candidates have redacted evidence, controls, scope references, and cleanup notes.
- Manual operator time is reserved for auth, OTP/CAPTCHA, phone/payment/KYC, account-control setup, tactical corrections, and final submit approval.

Secondary outcomes:

- CVE/freshness detectors become reusable platform modules.
- Lab proofs become bounded live-safe evidence patterns.
- Agent reviews improve attack paths and evidence quality without becoming conservative blockers.

## Operating Model

The platform runs in four layers:

1. Continuous intelligence
   - CVE advisories, H1 freshness, scope changes, program updates, and vendor advisories.
   - Output: ranked candidates and detector work items.

2. Detector and bundle execution
   - Passive fingerprint/version/exposure detectors for known CVEs.
   - Practiced bounty bundles for access-control, lifecycle, metadata, upload-marker, and API/UI mismatch lanes.
   - Output: candidate records, proof requirements, and lane state.

3. Evidence and review
   - Redacted screenshots, request snippets, expected-vs-observed matrices, positive/negative controls, and stop-before boundaries.
   - Multi-agent review is advisory unless it identifies a concrete blocker.
   - Output: report packet or explicit PARK/KILL reason.

4. Operator inbox and submission
   - Operator receives short decisions, not raw logs.
   - Operator approves scarce platform actions, human verification, credential gates, and final report submission.

## Safety and Authorization Boundary

This is an authorized-only platform.

Allowed by default:

- Local labs and owned test instances.
- Passive public fingerprinting and version/exposure checks when allowed by program policy.
- Owned-account/object evidence collection inside explicit scope.
- Report drafting with redaction.

Human-gated or policy-gated:

- Auth, OTP, CAPTCHA, phone, payment, KYC, OAuth/integration setup, API-token creation, mailbox/channel setup, and final report submission.
- Any active scanning, fuzzing, callbacks/OAST, exploit proof, or state-changing request on live targets.

Never store in repo/memory:

- Passwords, OTPs, cookies, tokens, verification links, API keys, private customer data, non-owned secrets, or raw sensitive program details.

## Repository Shape

- `platform/` — core automation engine: detectors, recon, pipeline, bounty, intel, inbox, agents.
- `programs/` — per-target live bounty lane state and evidence.
- `modules/` — reusable proof bundles and detector modules.
- `labs/` — local vulnerable apps, proof validation, and lab runbooks.
- `reports/` — generated reports, templates, and submission drafts.
- `handoff/` — compact rolling IPC and current strategy queue.
- `docs/policy/` — safety, scope, model, review, and evidence policies.
- `docs/runbook/` — operator and agent operating procedures.
- `docs/charter/historical/` — original expert-workbench charter and retired structures.
- `notes/` — learning, defense, and Obsidian project notes that support but do not drive the platform.
- `intelligence/cve_briefs/` — generated CVE brief archive.

## Decision Vocabulary

- EXECUTE: preconditions are met and the proof stays within authorization and safety boundaries.
- PASSIVE-ONLY: only version, exposure, or metadata checks are allowed on live targets.
- PARK: useful hypothesis but missing controls, scope clarity, or operator setup.
- KILL: low-value, out-of-scope, unsafe, duplicate-prone, or not worth revisiting soon.

## Current Strategic Bias

Prioritize first reportable bounty over broad learning:

- Fresh/new/updated H1 programs and mid-sized SaaS with clean self-serve controls.
- Latest CVE detector lanes only when they can be reduced to safe passive/live checks and local lab proof.
- Access-control and lifecycle bundles when owned A/B/C accounts and objects exist.
- Script-first evidence, not manual one-off browsing.

Learning assets remain valuable, but they support the platform; they are not the primary workflow.

# Repository Hygiene Policy

Status: active policy
Scope: `<private-workspace>` authorized bug bounty automation platform

## Decision

Do not reset this repository and do not restart from a blank repository as the default cleanup strategy.

This repo is messy, but it contains authorization boundary history, lane state, scope/policy decisions, platform code, tests, evidence pointers, and operator workflow context. Those are more valuable than a visually clean tree. Cleanup must preserve provenance first, then reduce noise.

Default cleanup strategy:

1. Freeze and inventory.
2. Classify files by role.
3. Split work into reviewable commits.
4. Quarantine uncertain material instead of deleting it.
5. Promote only tested/policy-backed artifacts to active project truth.

A new repo or clean-room copy is allowed only as a staging/migration aid, not as the source of truth, unless the operator explicitly approves a repository reset.

Allowed use cases:

- build a temporary clean staging copy for structure comparison;
- prototype a reduced tree shape without secrets/evidence;

Not allowed by default:

- replacing the main repo while leaving scope/policy/evidence history behind;

Do not run live scans or target-touching automation during cleanup.

### Phase 1 — Classify

Classify every modified/untracked/deleted path into one bucket:

- keep active truth;
- keep platform code/test/schema;
- archive historical/reference;
- quarantine uncertain/transient;
- ignore generated/local;
- delete only with explicit operator approval.

setting/local/quarantine/<YYYYMMDD>/

## What NOT to Build Next

以下類型不要產出，會增加 process gravity：

- 新的 multi-party review / decision policy 變體
- 新的 schema / contract（除非有具體 module 即將用）
- 額外 OSS Recon Gate 拓展
- 任何 governance-first 抽象層
- 額外 review tier 細分
- 新的 tactical preview / risk rebalance / freedom direction 變體
- 任何在 `handoff/` 根目錄的 root-level 新檔

如果你（任何 agent）覺得需要寫上面任一類，**停下來，把該想法加在本檔 Backlog 後**，等 P1-P5 完成且 operator_inbox 跑通後再評估。
