# picoCTF Rolling My Own: MD5-selected generated shellcode

Use this reference for reverse/crypto-ish CTF challenges where a custom password checker hashes chunks of input, selects digest bytes, and executes or compares the selected bytes.

## Recognition pattern

Indicators:

- Challenge says it "doesn't need to store the password" or uses a custom password checker.
- Binary imports `MD5_Init`, `MD5_Update`, `MD5_Final` or another hash API.
- Binary imports `mmap` and requests RWX memory (`PROT_READ|PROT_WRITE|PROT_EXEC`).
- User input is mixed with embedded constants/salts, hashed in fixed-size chunks, and selected digest bytes are copied into a small executable buffer.
- Generated code is called with useful registers already set, such as `rdi` pointing to a success/check function.

## Workflow

1. Stay in blind mode if requested: do not search for writeups. Use only the prompt, artifact, local tools, docs, and official remote endpoint.
2. Reverse the transform exactly:
   - input length and truncation behavior;
   - per-block constants/salts;
   - hash chunk size;
   - selected digest offsets;
   - byte ordering into the final candidate/check buffer;
   - register state at the generated-code call.
3. Look for the shortest useful generated code, not necessarily a full direct call:
   - If a success/check function pointer is passed in a register, consider jumping into its success path (`add rdi, offset; jmp rdi`) instead of materializing the expected magic argument.
   - If only a compare result is needed, consider short branch/return stubs.
   - Bytes after an unconditional `jmp`/`ret` can often be arbitrary, lowering preimage constraints.
4. Convert the desired code into per-hash-block constraints. Prefer prefix constraints when later bytes are unreachable.
5. Brute force each independent small input chunk locally. For binary passwords, allow non-NUL/non-newline bytes where `fgets`/`strlen` constraints permit them, and submit with a socket script rather than an interactive terminal.
6. Verify before remote submission:
   - reimplement the transform;
   - confirm generated bytes disassemble or match the intended machine-code prefix;
   - ensure no NUL before the bytes needed by `strlen` unless the checker intentionally permits it.
7. Use Kali for remote `nc`/socket interaction and keep it to the smallest final submission.

## Concrete Rolling My Own pattern

The known binary used four 8-byte salts:

```text
GpLaMjEW
pVOjnnmk
RGiledp6
Mvcezxls
```

It built 4 chunks of `password_4_bytes + salt_8_bytes`, MD5-hashed each 12-byte chunk, and copied selected 4-byte digest windows using offsets `[8, 2, 7, 1]` into a 16-byte RWX buffer.

At generated-code entry, `rdi` pointed to the flag/check function. Instead of generating a full `check(0x7b3dc26f1)` call, a short success-path jump was enough:

```asm
add rdi, 0x42
jmp rdi
```

Bytes:

```text
48 83 c7 42 ff e7
```

This required:

- block 0 selected bytes exactly `48 83 c7 42`;
- block 1 selected prefix `ff e7`;
- remaining bytes arbitrary because the jump leaves the generated buffer.

A validated password hex was:

```text
0e52a768010101e14141414142424242
```

Generated code prefix:

```text
4883c742ffe7
```

Remote flag obtained:

```text
picoCTF{r011ing_y0ur_0wn_crypt0_15_h4rd!_7b560461}
```

## Pitfalls

- Do not constrain all 16 generated bytes if an early `jmp` or `ret` makes the suffix unreachable.
- Do not assume printable passwords; use socket scripts and record hex when control bytes are valid.
- Avoid NUL bytes in the password before the used region when the checker calls `strlen` after `fgets`.
- If local execution fails because of an old library such as `libcrypto.so.1.1`, still verify by faithfully reimplementing the transform and then doing one minimal authorized remote submission.
- Be transparent whether the solve was blind or assisted; this pattern was solved blind using local reversing and brute force only.

## Bug bounty mapping

This pattern maps to custom client-side/thick-client password, license, or challenge-response checkers:

- Evidence should include transform pseudocode, constants/salts, selected digest/decision bytes, payload hex, and local reimplementation verification.
- Candidate findings should remain candidate until the bypass is tied to a real authorization boundary and impact.
- Binary/control-byte payloads require safe reproduction scripts, not just pasted strings.
