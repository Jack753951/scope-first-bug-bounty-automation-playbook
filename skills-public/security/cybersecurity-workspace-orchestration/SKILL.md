## Operating Posture

Default to authorized, repeatable, explainable, and defensible work. Treat scanner output as triage until manually verified with evidence, impact, remediation, and retest notes.

When the operator has already built an intentionally vulnerable local lab and explicitly asks to use it, do not keep retreating to offline-only planning. Actively use the lab within fixed, bounded controls: documented local target only, pre/post health checks, small fixed request lists, request caps/timeouts, redacted JSONL observations, and candidate-only semantics. The point of the lab is to calibrate future automation; after a manual/orchestrated run, productize the pattern into reusable adapters/modules rather than leaving it as a one-off.

When the operator explicitly pauses over-broad internal safety/tier/profile controls for the current learning stage, do not respond by adding another layered policy before acting. For the authorized disposable local靶機 / intentionally vulnerable app, default to script-first learning: use mature tools and existing scripts, cap obvious runaway behavior, keep artifacts local, recover if needed, and record candidate-only lessons. This pause does not remove external/legal/abuse red lines: no unauthorized public targets, malware, stealth persistence, real credential theft, real exfiltration, unauthorized pivoting, evasion, automatic submission, or automatic confirmed-finding promotion. See `references/learning-stage-local-lab-safety-pause.md`.

When the operator corrects the architecture toward a script-first workflow, do not keep pushing a contract/profile/validator-first path. Make preview/recon context, script inventory, situational script bundles, result review, and modularization the main operator-facing loop; keep scope gates and heavier contracts as guardrails for risky or report-impacting boundaries. See `references/script-first-context-driven-module-loop.md`.

When the user asks for the long-term goal, current phase, project status, or “where are we now?”, answer in the user's preferred language and make the roadmap explicit: long-term goal, current phase/checkpoint, remaining sub-phases or gates, what is intentionally not authorized yet, and the safest next action. Keep it concise enough to be useful in a terminal; do not bury the phase/status answer under implementation details.

Do not provide or run workflows for stealth, persistence, credential theft, malware, destructive actions, evasion, unauthorized access, or out-of-scope target interaction.

## Workstream Classification

Classify every request before acting:

1. `build/automation` — scripts, parsers, scope validators, dry-run modes, dependency checks, tests, templates, repo structure. Primary worker: Codex.
2. `strategy/knowledge` — learning plans, technique libraries, documentation cleanup, protocol notes, roadmap design. Primary worker: Cowork.
3. `threat intelligence` — CVE watch, CISA KEV, vendor advisory comparison, patch priority, exploit maturity, mitigation/detection notes. Hermes gathers current primary sources; Cowork synthesizes; Codex automates if useful.
4. `defense/reporting` — hardening baselines, firewall/logging guidance, IR playbooks, finding templates, executive summaries, remediation and retest wording. Cowork handles narrative; Codex handles templates/tools.
5. `threat hunting / SIEM lab` — read-only investigation over authorized lab telemetry such as Splunk. Hermes can build IOC pivots, timelines, process trees, decoded command evidence, and a report draft; keep it explicitly non-target-touching and convert the closeout into platform/reporting lessons. Use `references/splunk-threat-hunt-lab-workflow.md`.
6. `incident response` — suspected compromise or live incident. Preserve evidence first, write an IR case note, avoid cleanup-only thinking, and do not destroy forensic artifacts.

## Authorization And Scope Gate

Before any active scan, exploitation, brute force, callback, or target-touching automation, confirm the task includes one of:

- local lab or intentionally vulnerable app
- CTF / training platform
- user-owned asset
- written client authorization
- explicit bug bounty scope

If scope is unclear, pause and ask for scope instead of providing commands, payloads, or automation. For bug bounty/client work, prefer asking for: program/client name, in-scope assets, out-of-scope assets, allowed techniques, rate limits, testing window, and reporting rules.

Treat "passive-ish" remote CVE modules as target-touching when they send any request to external assets, even if they only read headers or public metadata. Before wiring such modules into automation, require scope enforcement, dry-run, conservative rate/concurrency defaults, and a triage-only output taxonomy. Local configuration audits on owned/authorized hosts can be handled as defensive audit tools, but still label their authorization assumptions clearly.

### CTF / Training Challenge Workflow

When the user provides a CTF/training URL, instance, attachment, or leaked source/database files, keep target interaction narrowly scoped to the explicit challenge and inspect local artifacts before live probing. Prefer minimal verification over noisy brute force: read the page/source/JS/WASM/database, solve or transform artifacts offline where possible, then use the smallest number of live requests needed to verify.

For this user's cybersec workspace, external CTF/lab website interaction and security/tool execution should prefer Kali, while Windows stays the control plane for orchestration, Obsidian, and repo-local artifacts. Store throwaway challenge downloads/solvers under git-ignored local paths such as `setting/local/<challenge-slug>/` and save durable technique lessons in Obsidian.

Use a lightweight output-side review gate for CTFs: if a candidate flag has abnormal format, the UI/checker result is ambiguous, WASM/binary logic uses `strcmp`/`strlen` or embedded NUL bytes, multiple keys decode plausible files, QR/OCR output is low-confidence, or the next step would send payloads/fuzz/scan, perform a second review before reporting the answer. See `references/ctf-web-reverse-workflow.md` for the class-level workflow and Web/JS/WASM pitfalls.

For Flask CTF apps, explicitly check whether sensitive state such as OTP/2FA values is stored in Flask's default client-side `session`. Default Flask session cookies are signed for integrity but not encrypted, so their JSON payload can be decoded and read without the secret key. If `otp_secret` or similar values are stored there, decode the cookie after login and submit the OTP once instead of brute-forcing the 2FA endpoint. See `references/ctf-flask-session-and-leaked-db.md` for the concise workflow and decode sketch.

5. ### Sustained Review Loop Preference

When the operator wants a durable multi-agent process, keep this loop by default:

1. Cowork/Claude proposes or reviews strategy.
2. Codex implements minimal safe patches.
3. Cowork/Claude independently reviews Codex's result.
   - Require a verdict (`ACCEPT` or `ROUTE BACK`).
   - Separate blocking issues from non-blocking improvements.
   - Include strategic third-party recommendations for future phases.
   - Evaluate architecture fit against the operator's long-term goals: extensible, update-friendly, modular, systematized, safely gated, and agent-assisted.
4. Hermes verifies and updates handoff state.
5. Only then proceed to the next phase or route blocking items back to Codex.

Do not run every small patch through the heaviest review path by habit. Use the current repo policy and the lightest validation that protects real authorization/safety/data-loss boundaries. In the Cybersec Lab, historical review tiers and multi-party approval gates are superseded: external workers are advisory unless they identify a concrete hard-stop blocker.

### Growth-First Review and Hard-Stop Governance

When a cybersecurity workspace starts accumulating many small review files or the user says the process is slowing growth, simplify toward hard stops plus focused validation rather than adding more tiers.

Use this default framing:

- Proceed by default for offline/local capability-library work, local-lab proof iteration, documentation cleanup, validators, fixtures, and established in-scope owned-account manual checkpoints when no concrete hard stop applies.
- Use Claude/Cowork/Codex review when it materially improves tactical coverage, architecture, evidence quality, or dissent capture.
- Treat reviewer output as advisory unless it names a concrete authorization/scope, safety, validation, malformed-consumed-contract, or data/capability-loss blocker.
- Preserve hard stops for ambiguous/missing scope, live scanners/fuzzers/DAST/exploits/callbacks/OAST/tunnels/proxies/pivots, OAuth/integration/webhook/channel/mailbox/API-token/billing/payment/KYC/scheduler/deployment/publishing/report-submission activation, secrets/credentials/cookies/tokens/OTP/phone/verification links/loot/customer data, destructive/persistent/stealthy/evasive/brute-force/resource-exhaustive/malware-like/uncontrolled behavior, and report-ready promotion/final submission.

Batch related slices only when it reduces overhead. Re-run external review only when assumptions materially change or a hard-stop boundary is near. For small follow-up recommendations, implement the narrow change, run focused validation, update handoff/accepted_changes when durable, and keep moving.

When a cluster of micro-slices is complete and the next step would cross a real hard-stop boundary, write a compact checkpoint and bounded next action. The checkpoint should name completed evidence, deferred recommendations, re-open triggers, validation, and explicit non-authorization of target/runtime changes.

Default proxy/pivot/transport posture remains strict: offline docs/plans are fine; any code that can open sockets, proxy, tunnel, callback, beacon, forward traffic, invoke external network tools, persist, or carry secrets needs explicit boundary review and often operator approval. Do not express this as old T-tier ceremony; express the concrete side effect and approval requirement.

### Periodic Third-Party Strategy Review

For long-running cybersecurity workspaces, periodically review the whole project and future strategy, not just individual patches. Default cadence: weekly. Include memory coexistence and cross-project contamination checks when the workspace shares a Hermes profile with other projects; see `references/memory-coexistence-governance.md`.

1. Hermes creates a current project snapshot under `handoff/periodic_reviews/YYYY-MM-DD/`.
2. Codex reviews engineering/systematization: architecture, code organization, schemas/contracts, tests, modularity, maintainability, and technical debt.
3. Claude/Cowork reviews strategy/product-security: roadmap fit, safety governance, bug bounty workflow quality, agent-assisted analysis, extensibility, and sequencing.
4. Hermes synthesizes shared recommendations, disagreements, immediate next actions, defer/avoid items, safety notes, and suggested phase order.

When the workspace has adopted a multi-party review policy, scheduled reviews should follow the current live policy rather than stale historical gates. In the Cybersec Lab this means growth-first hard-stop review: read `docs/policy/review_tiering_policy.md` and `docs/policy/multi_party_review_decision_policy.md`, identify concrete blockers, separate advisory recommendations, include capability-growth drift, and avoid claiming missing worker routes are blockers by themselves.

Give all reviewers the same concise evidence packet (job/repo/branch/PR, required files read, local command outputs, current phase/navigation, safety boundaries, original job instructions). If delegation is unavailable, times out, or a reviewer lacks evidence, the final report should name the missing role/evidence without converting that absence into a blocker unless the current policy requires it. See `references/scheduled-delegate-cross-review.md` for the concrete cron pattern, reviewer prompts, validation/commit/PR-comment sequence, and Traditional Chinese final-report shape.

This periodic review must not run scans, touch external targets, modify `config/scope.txt`, execute modules, publish reports, or store secrets. See `references/periodic-third-party-strategy-review.md` for the output layout, prompt template, and cron pattern.

7. Hermes performs the final verification/arbitration pass, updates handoff artifacts, and summarizes back to the human owner with workstream, authorization status, files changed, validation status, and remaining risks.

Keep Windows Hermes as the control plane for memory, skills, Obsidian, planning, and final synthesis. This is host/tool isolation, not anonymity: Kali may still share the same external egress. Scope gate remains mandatory before every active scan, exploit, brute force, callback, proxy-routed request, or target-touching automation.

For Phase 4A two-Kali labs, treat the red-team Kali and victim Kali as separate roles: red-team VM runs tooling, victim VM runs intentionally vulnerable apps. After setup/image pulls and snapshotting, the victim should retain host-only networking but disable NAT, file-transfer clipboard, drag-and-drop, and writable shared folders; bidirectional text clipboard can stay on temporarily when the operator explicitly wants command-copy convenience, but label that as controlled lab convenience rather than strict isolation. Verify app reachability from red-team Kali by the victim's host-only IP, not by `127.0.0.1`, and first check the red-team VM IP so the suggested victim static IP does not collide. If Host-only DHCP is off and a VM lacks a `192.168.56.x` address, manually assign a free lab IP rather than misdiagnosing the NIC as detached. A failed ping from the victim to the Windows host-only address can be a Windows Firewall ICMP artifact; verify actual service reachability with TCP/HTTP from Windows and red-team Kali before treating networking as broken. If VirtualBox reports only a transient shared-folder mapping, unmount it inside the guest and power off/recheck; permanent mapping removal alone may not clear the running VM state. For the first Juice Shop baseline, do not add proxychains by default: run direct host-only `curl`, `whatweb`, and single-known-port `nmap`, then save candidate-only evidence; use `references/phase4a-victim-lab-hostonly-juice-shop-control.md` for victim setup/recovery and `references/phase4a-juice-shop-baseline-calibration.md` for the exact baseline sequence.

When the operator asks for host/VM safety hardening or to put a lab VM into a controllable setup state, proactively apply safe local controls that do not need secrets or target interaction: inspect VirtualBox settings, confirm host-only + NAT networking, and set clipboard/drag-and-drop according to mode. For setup-control mode, bidirectional clipboard/drag-and-drop can be enabled temporarily when the operator needs to paste commands or configure the victim VM; keep clipboard file transfer off and state that this is not isolated attack mode. For hardening/attack mode, disable running-VM clipboard/file-transfer/drag-and-drop channels with `VBoxManage controlvm`, show Windows file extensions, verify Defender/Firewall state, and create git-ignored helper scripts for sudo/admin-only steps. For lab preparation, distinguish setup mode, setup-control mode, and attack mode: keep NAT available while downloading packages/Docker images, snapshot after tools/images are ready, then disable NAT/shared writable folders and convenience channels before aggressive testing. Do not claim you can install inside a VM merely because VirtualBox shows it running; first verify SSH or Guest Additions `guestcontrol` credentials. Do not pipe guessed/default passwords into `sudo -S`; if Kali sudo is needed, write a local helper under `setting/local/` and ask the operator to run it manually. See `references/host-vm-hardening.md` and `references/phase4a-kali-vm-lab-setup.md`.

## Extensible Architecture Direction

For cybersecurity labs intended to grow into reusable bug bounty or defensive automation platforms, steer implementation toward update-friendly, systematic architecture rather than one-off scripts:

- Keep scope/rule policy data separate from execution logic (`config/scope.txt`, `programs/<slug>/scope.json`, future module manifests).
- Build program scope in layers: schema/examples/docs first, then an offline standard-library validator, then a separate offline policy decision helper (`target + technique + mode` against program scope + global scope), and only later runtime recon integration.
- Treat the policy decision helper as a reusable contract before runtime wiring: default-deny, no network/subprocess target interaction, program `out_of_scope` precedence, global+program scope intersection, technique gates, automation/window gates, and non-zero exit on deny.
- Split runtime integration into reviewable phases: CLI/no-program regression surface first, a jq-free Python boundary wrapper second, and only then per-stage runtime wiring. Preserve no-program behavior as byte-identical zero side-effect behavior, and require fresh policy decisions for every target/stage rather than cached allows.
- Before building new platform primitives such as schemas, module manifests, or finding/evidence contracts, briefly inspect established open-source formats/tools (for example Nuclei templates, SARIF/OSV/DefectDojo concepts, bug bounty scope tooling), then explicitly decide whether to adapt or write a project-specific contract.
- Prefer versioned JSON schemas and stable CLI contracts for program scope, policy decisions, run manifests/execution ledgers, module manifests, findings, evidence, and reports.
- Treat vulnerability scripts/checks as future modules/plugins with declared metadata: stable module ID/version, technique tags, target types, risk, mandatory dry-run support, network requirements, target-touching posture, external tools, safety gates, and output schema references. Pair JSON Schema with a stricter semantic validator so unknown techniques, missing dry-run behavior, unsafe gates, duplicate tags/types, passive target-touching, and known technique/network mismatches all fail closed; see `references/module-manifest-contract.md`.
- When discussing offensive or pentesting scripts in a bug-bounty platform, avoid saying they are generically banned if the real issue is premature automation. Use a risk-tier policy: pentesting techniques can be supported only through authorization, scope/program rules, execution mode gates, candidate-only outputs, and review/operator approval; unauthorized/unscoped/unreviewed/automatically-confirming attack automation remains prohibited. See `references/active-testing-risk-tier-policy.md`.
- When discussing active/offensive bug-bounty techniques, avoid saying they are simply "excluded" or "banned" unless they are truly prohibited behavior. Prefer the risk-gated framing: valid pentesting techniques can become gated active-testing modules only after authorization, program rules, execution modes, risk tiers, review tier, candidate-only outputs, and operator approval are satisfied. Use `references/active-testing-risk-tier-policy.md` for the Tier 0-Tier 5 model and wording.
- Add a run manifest / execution ledger before building a real module runner: bind `run_id`, target, program/global scope hashes, policy decision artifact/hash, execution mode, modules, finding/evidence artifact references, and triage review state. Keep it offline-only until runner authorization gates exist.
- When introducing the first module runner, make it a dry-run-only offline planner: consume validated manifests and a pre-existing allow policy artifact, re-check path/hash/request identity and error/deny fields, emit a planned run manifest preview, and explicitly avoid module execution, subprocess/process launch, network access, target touching, findings, evidence, or callbacks. See `references/dry-run-module-runner-skeleton.md`.
- After the dry-run runner exists, the first Level 1 audit module should be a committed offline fixture under `modules/checks/<level-or-category>/<module_id>/module.json`: data-only manifest, `requires_network=false`, `network_access=none`, `target_touching=false`, no destructive/intrusive behavior, and no findings/evidence emission. Test both validator acceptance and runner planning of the committed fixture before adding discovery or execution. See `references/level1-audit-module-fixtures.md`.
- Add offline module discovery/profile selection only after at least one committed safe fixture exists. Discovery should scan only repo-local `modules/checks/**/module.json`, validate every discovered manifest, fail closed on malformed manifests and duplicate `module_id` values, select a conservative profile such as `audit-baseline` only when modules are dry-run-safe/no-network/non-target-touching, and pass selected manifests into the existing dry-run planning path without execution. Do not mix discovery mode with explicit manifest paths unless explicit paths receive the same repo-local containment/profile gate; failing closed on mixed mode is safer. See `references/offline-module-discovery-profile-selection.md`.
- Once discovery/profile selection exists, move profile rules into versioned repo-local data rather than relying only on runner hard-coding: `modules/profiles/<profile_id>.json`, `modules/profiles/INDEX.md`, `modules/_schema/module_profile.schema.json`, and a strict semantic validator. The runner should fail closed on missing/malformed/schema-invalid/mismatched/unsupported profiles and unsafe constraints, apply the same profile gate to discovery and explicit manifests, and bind `execution.profile_id` plus `execution.profile_sha256` into the dry-run preview. Prefer stable profile error codes over free-form prose. See `references/data-driven-module-profile-contract.md`.
- After profile-gated dry-run planning exists, define a small offline module I/O contract layer before any real executor work: versioned `module_input/1.0` and `module_result/1.0` envelopes, strict stdlib semantic validation, and a preview-only runner flag that emits input/result previews only after manifest/profile/policy/run-plan gates pass. Result previews must stay `not_executed`, `dry_run=true`, `target_touching=false`, and empty findings/evidence; do not import module code, spawn subprocesses, open network clients, or touch targets. See `references/module-io-contract-layer.md`.
- After module I/O previews exist, add a thin offline bundle consistency layer before any persisted ledger or executor work: validate a planned `run/1.0` plus parallel `module_input/1.0` and `module_result/1.0` arrays, match previews by `module_id` plus manifest hash, fail closed on missing/duplicate/extra pairs and run/module/policy/profile/program/target/hash/path/artifact mismatches, reject non-empty findings/evidence, and keep the validator standard-library-only with no module imports, subprocesses, network, targets, or persisted ledger writes. Defer preview hash/audit ledgers to a separate reviewed phase.
- When hardening the offline bundle consistency layer, preserve existing `errors`/`error_codes` while adding additive `error_details` objects with stable `code`, human `message`, precise `path`, `expected`, `observed`, and `module_id` where applicable. Route JSON-load failures and all `add_error` paths through the same helper so details are complete; include explicit tests for every important `BUNDLE_*` code, timestamp drift between `run.created_at_utc` and `module_input.run.created_at_utc`, and multi-module one-missing/one-extra module-ID mismatch cases. See `references/module-io-bundle-diagnostics-hardening.md`.
- After bundle diagnostics are stable, a separately reviewed offline preview persistence layer may write already-validated dry-run preview bundles under repo-local `runs/<run_id>/preview/`. Make it dedicated opt-in (not generic `--output`), require module I/O previews and explicit repo root, write only fixed JSON artifact names after all gates allow, reject unsafe run IDs/path escapes/symlinked parents/existing preview dirs, catch filesystem failures as structured deny, and preserve no module imports/subprocess/network/scanner/target-touching/finding/evidence behavior. Check symlinks on the unresolved lexical parent chain before `resolve()` as well as resolved containment. See `references/offline-preview-bundle-persistence.md`.
- After preview persistence exists, add a separate offline preview manifest contract before ledger/archive/agent-triage consumers rely on persisted bundles. Use a closed strict-version schema such as `preview_manifest/1.0`, bind only fixed dry-run JSON artifacts by safe relative path, SHA-256, size, and content type, exclude `preview_manifest.json` from its own artifact list, and provide a standalone stdlib read-only validator that fails closed on malformed JSON, duplicate keys, unknown fields, unsafe paths/run IDs, symlinks, missing/extra artifacts, hash/size drift, non-allow bundle consistency, and timestamp drift. Keep it T3 with OSS Recon Gate and independent review; do not wire it into execution or reporting. See `references/offline-preview-manifest-contract.md`.
- Candidate finding/evidence contracts must preserve triage-only semantics: automation may emit `candidate` / `needs_verification` records, but not confirmed findings; evidence metadata must be redacted, repo-safe, run-linked, integrity-hashed, and resistant to path traversal or sensitive key/value smuggling.
- After several offline platform contracts are in place, deliberately pause new schema/ledger/archive abstraction for at least one sprint and run an offline workflow-validation slice. Pick one low-risk Level 1 module such as `security_headers_baseline`, keep it fixture-driven and stdlib-only, generate candidate-only findings plus redacted human triage drafts, and require a candid Claude/Cowork direction review that may object to over-engineering. Do not wire runtime emission, runner/recon/profile/CI/scheduler, live target flags, or schema bumps during this sprint. Add explicit tests that reject live-target flags (`--target`, `--url`, `--host`, `--scope`, `--live`) and use AST-backed static checks for forbidden imports/calls rather than only regex scans; see `references/offline-workflow-validation-sprint.md`.
- When reviewing manifest/profile readiness against future active-testing policy, first produce a documentation-only crosswalk that maps current `module_manifest/1.0` and `module_profile/1.0` fields to policy concepts, identifies gaps, and freezes future vocabulary as non-contractual design memory. Do not add schema fields, fixtures, validators, runner behavior, discovery behavior, or any target-touching capability during the crosswalk slice; see `references/manifest-profile-policy-crosswalk.md`.
- When adding an offline bridge between recon policy artifacts and the dry-run runner preview, keep it tests-only first: create a temp HACKLAB, copy the recon-emitted allow `policy_boundary_*.json` into temp `runs/<run_id>/policy/decision.json` inside the test harness only, invoke the existing runner in `--mode dry-run`, assert preview/non-execution semantics plus fail-closed mismatch cases, and explicitly defer runtime bridge/auto-copy/CLI flags. If a delegated worker times out and RED evidence is incomplete, record the caveat in a named result file, rerun focused/adjacent/full validation directly, and require a lightweight independent safety review before runtime bridge discussion. See `references/offline-recon-to-runner-bridge-coverage.md`.
- When CTF/lab work is used as calibration, the success metric is not only the flag. Require a short closeout that converts the exercise into bug-bounty workflow learning: solving mode/source transparency, root cause, evidence shape, false-positive traps, candidate-finding analogy, manual verification checklist, report-readiness blockers, and any platform artifact to update. If the user asks for another CTF after a solve, offer `blind training` versus `assisted solve` modes and prefer Web/source-review targets when the stated goal is bug-bounty workflow improvement.
- When an offline/dry-run bug-bounty platform is nearly ready but not safe for real targets, insert a controlled-lab semi-automated calibration phase before any real program private beta. Keep it after dry-run/local MVP closeout and before one-program beta planning; require explicit lab scope, concrete activation-boundary review, deny-by-default non-lab tests, rate/timeout/kill/audit controls, candidate-only outputs, human verification, evidence redaction, false-positive review, and report-readiness observations. See `references/controlled-lab-semi-automated-calibration.md`.
- For SIEM/threat-hunting labs, treat the hunt output as a bug-bounty/reporting calibration artifact: state scope/mode, hypothesis, confirmed telemetry, unconfirmed follow-on activity, SPL pivots, affected entities, containment/detection recommendations, and automation lessons. For package-manager supply-chain hunts, explicitly capture package version, lifecycle script, package-manager-descended process tree, decoded interpreter payload, network IOC, persistence mechanism, and host/user expansion. See `references/splunk-threat-hunt-lab-workflow.md`.

When the project starts drifting into CTF scaffolding, close CTF work as calibration support and redirect the next slice back to authorized bug-bounty workflow value. A safe sequence is: candidate findings -> trial candidate review packet -> trial gap/action report -> offline verification plan/checklist -> report-readiness gate/classifier. Keep each consumer stdin/stdout or fixture-only, triage-only, deterministic, and explicitly non-promotional until multiple consumers prove the contract shape. Verification-plan consumers should use stable checklist item codes, fail closed on unknown review states/gap codes/count anomalies/duplicate gaps, and avoid report prose. See `references/candidate-review-packet-gap-consumer.md`.
- For a local/intentionally vulnerable web lab used to test the "full bug-bounty flow," do not stop after scanning or candidate gating. Exercise the whole chain: baseline recon -> model review -> bounded script selection -> active lab scripts -> second model review -> low-risk verifier -> candidate packet -> gap/verification/report-readiness consumers -> lab-only pentest report draft. Keep `/ftp/`-style exposed-directory checks metadata-only first, treat SPA fallback as a common false-positive trap, and explicitly state whether the flow is productized or still Hermes-orchestrated. When an output-side review approves manual content-class verification, use a fixed GET-only candidate list, request cap, pre/post health, metadata/hashes/short redacted snippets only, and cautious `metadata exposure` wording before marking a lab report draft ready. When the operator asks to test the project upper bound or allows aggressive/destructive scripts, split that into a separate isolated-snapshot slice with recovery, timeout, request caps, kill switch, audit log, redaction, and a fresh multi-party review; do not mix destructive testing into the standard report rehearsal. For the first isolated aggressive-lab execution slice, prefer a bounded content-discovery rehearsal with a fixed small `HEAD` path list, pre/post health checks, metadata-only JSONL, and output-side false-positive review before any body-aware follow-up. For lab-only report review, preserve third-party worker usage/model artifacts when available and label the final state as lab-only accepted, revise, or blocked. See `references/phase4a-juice-shop-end-to-end-bugbounty-flow.md`, `references/phase4a-victim-lab-control-and-bounded-report-flow.md`, `references/phase4a-bounded-content-discovery-rehearsal.md`, `references/phase4a-content-class-to-lab-report.md`, and `references/lab-report-review-and-aggressive-gate.md`.
- When productizing that rehearsal, first build a guarded `plan_only` lab workflow runner rather than an unattended executor. Require explicit lab mode, reject public targets, emit fixed artifacts for scope gate, baseline recon, model review/script selection, active script plan, candidate packet, gap report, verification plan, report-readiness gate, and lab report draft plan. Keep actual max-lab execution separate/manual until an execution adapter has timeouts, health checks, redaction, audit logs, and recovery controls. See `references/phase4a-lab-workflow-runner-and-maxlab.md`.

Cover services such as Apache, Tomcat, TLS/OpenSSL, HAProxy, Envoy, and Traefik with fixed safe probes, plan-only defaults, explicit lab approval for runnable scripts, public-target fail-closed behavior, local artifacts, and candidate-only wording. Update script inventory, bundle docs, and accepted changes, and only promote stable checks into modules after review. See `references/lab-service-baseline-bundle-adapter.md`.

Start with mature structured-output tools such as ProjectDiscovery `httpx`/`katana`; treat `nuclei`, `ffuf`, ZAP active scan, and random exploit PoCs as later/higher-risk phases with allowlists, static review, and health controls. Importer outputs must remain non-promotional observations, not findings. See `references/phase4a-external-tool-adapter.md`.
- When turning a successful local-lab web metadata rehearsal into Phase 4B modules, create Level 1 data-only manifests before writing an executor. Keep checks such as directory-listing metadata, robots/security.txt metadata, API-docs metadata, dependency-manifest metadata, and CORS metadata as non-network/non-target-touching manifest records first; validate every manifest and run project review. A later lab adapter must require explicit lab approval, local/private target restriction, fixed path/origin lists, request caps, pre/post health, JSONL observations, no file downloads, no recursive crawl, no exploit payloads, and no finding promotion. See `references/phase4b-level1-metadata-module-bundle.md`.
- When planning OWASP Top 10 coverage, do not assume it is annual. Track official web-application release editions (2003, 2004, 2007, 2010, 2013, 2017, 2021, 2025) for traceability, keep the currently implemented taxonomy stable until mapping/migration review, and treat historical/latest release coverage as catalog/planning only until each module has scope, safety tier, run card, and candidate-only output gates. See `references/owasp-top10-release-coverage-planning.md` and `references/owasp-top10-traceability-matrix-and-2025-migration.md`.

### Post-Scan Agent Review

For non-trivial scan/module runs, do not rely only on scripts or raw scanner output. Treat module output as structured `candidate` findings, have Hermes verify scope/run/audit integrity, then route evidence to Claude/Cowork for false-positive review, impact reasoning, manual verification steps, and report-quality analysis. Codex fixes automation defects found during review. Use `references/post-scan-agent-review-workflow.md` for the artifact contract and lifecycle.

Add an output-side review decision after every verification script/tool run. Low-risk informational outputs with clear scope, complete evidence, no sensitive data, and no exploitability claim may be archived or drafted directly. Require second review before treating results as confirmed when outputs involve SQLi/XSS/SSRF/RCE/auth bypass/IDOR/file read/upload/deserialization/cloud metadata, high/critical scanner hits, possible secrets/tokens/PII, low-confidence high-impact claims, second-stage payloads, or report language that would assert a confirmed vulnerability. For CTF/reverse tasks, also second-review abnormal flag format, ambiguous checker behavior, C-string/NUL traps, and ambiguous decoded media.

- Re-read relevant project context files (`.hermes.md`, workflow docs, scope examples) when present.
- Validate JSON/YAML/scope files touched by the task.
- Run compile/test/lint checks when obvious and safe.
- Run the project's review/preflight wrapper when available.
- Confirm no worker lock/temp state is left behind. - For dirty working tree cleanup after multi-agent cybersec work, classify artifacts before acting: modified rolling files, durable code/tests/fixtures, durable handoff/review/report records, rolling handoff pointers, and obvious temp garbage. Use explicit staging lists, never `git add .`, and avoid deleting `reports/`, `handoff/`, `modules/`, scan outputs, or review artifacts just to quiet status. See `references/kali-shared-repo-and-dirty-tree-hygiene.md`.
- Confirm handoff files reflect the final state rather than stale blocked notes.
- For Codex-implemented non-trivial changes, confirm an independent Cowork/Claude review was run or explicitly skipped by the operator.

## Pitfalls

- Do not over-rotate into offline-only planning after the operator has created and authorized a local intentionally vulnerable lab. If the user explicitly says to use the target/靶機, run bounded host-only lab probes with health checks and caps, then turn the result into reusable adapter/module capability. Preserve safety gates, but do not waste the lab by only writing plans. For low-risk Tier 1/Tier 2 host-only checks, prefer a lab fast lane: one-time boundary confirmation, immediate bounded execution, output-side review, and escalation only when moving to payloads/scanners/credentials/callbacks/destructive behavior; see `references/phase4b-lab-fast-lane-get-only-adapter.md`.
- Do not force-delete or take ownership of unknown locked recon/temp directories in a cybersecurity workspace merely to quiet `git status`; they may contain evidence or sensitive runtime artifacts. First mitigate with a narrow ignore rule and require elevated operator inspection before deletion.
- Do not save transient target details, PR numbers, or scan results as durable memory. Persist reusable workflow lessons in skills and session-specific details in `references/` files.
- Do not write a finding as confirmed solely from automated scanner output. Add manual verification/evidence/retest expectations.
- Do not let CVE-specific report templates hard-code unverified severity, affected versions, or exploitability claims. Verify primary sources before presenting them as facts.
- Before wiring offline policy decisions into runtime recon scripts, do not rely on bash parsing nested JSON with grep/sed or assume `jq` is available. Prefer a Python stdlib boundary wrapper that handles timeout, contract validation, atomic artifact writes, and shell-readable status. Invoke that wrapper as part of the security perimeter: fixed repo-owned path, no production environment override, isolated Python mode (`-I`) to block `PYTHONPATH/sitecustomize.py` forgery, and isolated Python mode again for any child helper subprocess the boundary launches. Validate artifact path containment, schema, boundary status, request fields, embedded decision object, boundary audit event, and current source-file hashes before treating stdout `allow` as sufficient. Parse boundary timeout env vars with strict canonical integer validation (for example reject `099`, `000`, huge integers, and out-of-range values before shell arithmetic); see `references/program-policy-artifact-validation.md`.
- Do not treat `--policy-mode dry-run` as authorization for live execution. It should only be valid when the runtime is also in dry-run mode; target-touching execution requires planned/live policy modes plus program automation/window/blackout checks.
- Do not let automation-generated schema records imply confirmed vulnerabilities. Candidate finding/evidence outputs must stay triage-only until manual/agent verification, evidence review, impact analysis, remediation, and retest steps are complete.
- Do not trust metadata redaction based only on forbidden key names. Also scan scalar values case-insensitively for sensitive request/response/header/cookie/token patterns, and keep JSON Schema regex behavior aligned with the stricter validator so schema-only consumers do not accept bypasses.
- Do not validate only the first finding in a run bundle. Multi-finding runs must cross-check every finding against the evidence set and the run manifest; otherwise later findings can smuggle missing evidence, mismatched hashes/kinds, or wrong `finding_id` values through a single-finding happy path.
- Do not allow duplicate IDs in ledger-like arrays (`modules[].module_id`, `artifacts.findings[].id`, `artifacts.evidence[].id`) or manifest arrays (`supported_target_types`, `technique_tags`). Dictionary construction can silently collapse duplicates and hide contradictory paths or hashes; semantic validators should fail closed on duplicates.
- Do not assume a Python semantic validator is enough for safety if other consumers may use JSON Schema directly. For module manifests and similar contracts, add unsafe fixtures that verify both the strict validator and the schema reject known technique/network mismatches and passive target-touching where expressible.
- Do not let a dry-run module runner become a hidden executor. The first runner should read manifests and policy artifacts as data, re-validate policy identity/error/deny/helper fields, and emit planning output only. Treat subprocess/process launch, network clients, dynamic imports of module code, `eval`/`exec`, finding generation, evidence capture, callbacks, or target-touching code paths as blockers until explicit live-execution gates exist; see `references/dry-run-module-runner-skeleton.md`.
- Do not expose inert future-facing flags without a warning. If `--program`/`--policy-mode` are accepted before per-stage policy decisions are wired, make the UX/audit text explicit that only the existing global scope gate is active.
- Do not rely on permissive shell matching for security-sensitive CLI enums. Force lowercase or exact-match validation for modes such as `dry-run`, `planned`, and `live`.
- Do not let module I/O previews weaken the dry-run runner boundary. Preview contracts should document future invocation/result shapes only; they must not import module implementations, spawn processes, invoke scanners, open network clients, emit real findings/evidence, or bypass existing profile/policy gates.
- Do not keep adding schemas, ledgers, archive indexes, or generic platform contracts after the project has enough offline scaffolding to test a real workflow. Over-engineering at this stage hides the more important question: can a safe fixture/local-lab observation become a useful candidate finding, triage draft, reviewer critique, and report-ready note without weakening scope gates?
- Do not let contract/profile/preview machinery become the operator-facing workflow when the user wants script-first modularization. If the user asks where scripts went or says the safety structure is too heavy, inventory practical scripts, define context-driven script bundles, execute/review a small authorized bundle, and modularize what works. Use heavier contracts only as guardrails where risk/report integrity requires them; see `references/script-first-context-driven-module-loop.md`.
- Do not answer an explicit learning-stage pause request by replacing one broad safety system with another detailed tier/profile system. For an authorized disposable local lab, act script-first with existing tools/scripts, then record candidate-only lessons and bundle the useful workflow. Keep external/legal/abuse red lines intact; see `references/learning-stage-local-lab-safety-pause.md`.
- Do not let CTF scaffolding become the main roadmap in this user's bug-bounty automation workspace. Treat CTF artifacts as calibration only unless they directly improve the authorized bug-bounty candidate -> review -> verification workflow.
- Do not let stdin-only review/gap consumers accept positional CLI arguments. Even a harmless-looking positional like `example.com` is a target affordance; reject all CLI args with structured JSON errors unless a later reviewed phase explicitly designs an argument contract.

- Do not stop a requested end-to-end bug-bounty-flow calibration at scanner execution or candidate triage. If the user asks for a complete test, include report-readiness gates and a lab-only pentest report draft so reporting quality is calibrated too.
- Do not treat Juice Shop/Nikto high-impact-looking paths as findings without content-class verification. Many paths resolve to SPA fallback HTML; verify route/content class before escalating. In bounded content-discovery rehearsals, repeated `200 text/html` with the same content length is usually a fallback signal, not proof that `/.git/HEAD`, `/package.json`, `/server-status`, Swagger, admin, or debug resources exist.
- Do not treat `Access-Control-Allow-Origin: *` as a CORS vulnerability when credentials are absent; record it as hardening/non-finding metadata unless a concrete cross-origin credentialed impact is verified.
- Do not let tiny ffuf/gobuster/nuclei/info-template lab output become a confirmed finding. For SPA apps, compare status, length, content type, title, and fallback baselines before escalating; keep `/api-docs/`, `/rest`, `/api`, and `/administration` style routes as metadata/candidates until content class and impact are manually verified.
- Do not invent annual OWASP Top 10 coverage. OWASP Top 10 web-application editions are release-based, not calendar-year-based; include official releases for traceability, but map/migrate categories offline before changing implemented modules or runtime tests. A release traceability matrix should remain non-authorizing by construction: current taxonomy unchanged, category mappings confidence-labelled, and `runtime_authorization: false` on every row until a later reviewed runtime gate exists.
- Do not interpret a local-lab crash or availability loss as permission to keep escalating. Stop active steps, verify route/VM/container state, recover only because it is an owned lab, record the event, and use it as evidence that unattended execution needs health checks, timeouts, kill/recovery controls, redaction, and audit logging before real bug-bounty use.
- Do not recursively download or inspect sensitive-looking `/ftp/` artifacts during first-pass lab calibration. Record metadata only unless the operator explicitly approves a narrower manual review step.
- Do not let an offline preview manifest become a generic artifact index or implicit execution authorization. It should bind only the fixed dry-run preview JSON artifacts and should explicitly exclude itself; downstream consumers must fail closed on unknown fields, unknown artifacts, path/hash/size drift, and version mismatch.
- Do not treat CTF validation as permission for arbitrary external probing. Use it only on explicit CTF/training/local-lab targets, record the training scope, keep findings as candidate until manually verified, and use the session to validate workflow artifacts/evidence discipline rather than to bypass scope gates.
- In CTF web challenges with leaked Flask source/data, do not default to endpoint brute force when local analysis reveals client-readable session state. Check whether OTP/2FA values are stored in Flask's signed-but-not-encrypted session cookie and decode it locally before trying high-volume OTP guesses.
