# Quality Checklist — run before delivering

Pass/fail. If anything fails, fix it before showing the user.

---

## Hero
- [ ] Headline names a concrete outcome (not a vague aspiration).
- [ ] Headline is ≤ 12 words.
- [ ] Subheadline names the target audience or the mechanism.
- [ ] One CTA, prominently placed.
- [ ] No company-centric language ("Welcome to…", "We are…").

## Value section
- [ ] 3–5 bullets, no more.
- [ ] Each bullet leads with the benefit, not the feature name.
- [ ] No technical jargon the target audience wouldn't recognise.
- [ ] At least one bullet quantifies the benefit (hours saved, $ earned, % reduced).

## Proof
- [ ] Some form of proof present (real testimonials, numbers, case studies, or `[PLACEHOLDER]` marked clearly).
- [ ] No fabricated testimonials, names, or numbers.
- [ ] If using member count, it matches what the user told us.

## CTA
- [ ] Single primary CTA repeated, not 3 different competing actions.
- [ ] CTA verb is benefit-led ("Claim my seat", not "Submit").
- [ ] Price visible near the button.
- [ ] Risk reversal (guarantee / cancel-anytime / free trial) under the button when applicable.

## FAQ
- [ ] 4–7 questions.
- [ ] Includes objections from discovery (time, skill level, ROI, fit).
- [ ] Answers are 1–3 sentences each.
- [ ] No question is purely cosmetic ("What is X?" when X is the page title).

## Consistency
- [ ] Tone matches what the user requested (or stated default if user didn't specify).
- [ ] Claims match price level — no "scale to 8 figures" on a $30 ebook.
- [ ] Scarcity claims (if any) are real, not "limited spots!" boilerplate.
- [ ] Mechanism (if present) is plausibly connected to the outcome.

## Format
- [ ] Output is plain Markdown, ready to paste.
- [ ] Section breaks are obvious (clear headings, not just bold lines).
- [ ] No `[PLACEHOLDER]` markers left without explanation of what goes there.
- [ ] No copy that reads like AI ("In today's fast-paced world…", "Unlock the power of…", "Game-changer", "Synergy").
