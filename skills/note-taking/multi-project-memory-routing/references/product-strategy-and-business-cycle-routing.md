# Product Strategy and Business-Cycle Routing After Architecture Extraction

Use this when a new or reshaped product has already adopted process patterns from source projects and the next user ask is to define the long-term goal, subscription/business model, phase goals, or operating cycle.

## Pattern

1. Re-read the target project's authority layers before giving strategy:
   - project charter / README / architecture notes;
   - active strategy queue / current handoff;
   - any prior long-term goal or roadmap artifacts;
   - Obsidian project index or strategy note if the project uses one.
2. Keep the answer target-domain first. Source projects remain process references, not product truth.
3. Frame the product as an operating loop rather than a feature list:
   - input/context;
   - signal discovery;
   - packet/plan generation;
   - asset or work production;
   - QA / approval / safety gates;
   - controlled activation or manual export;
   - analytics / evidence ingestion;
   - learning report;
   - next-cycle recommendation;
   - reusable module/playbook promotion.
4. If monetization is requested, separate assumptions from validation:
   - pricing tiers;
   - ARPA / gross margin assumptions;
   - CAC / churn / payback assumptions;
   - break-even customer counts;
   - conservative/base/aggressive scenarios;
   - explicit warning that the math is not market validation.
5. Route durable outputs project-locally:
   - write a named repo handoff/strategy artifact for the detailed recommendation;
   - mirror or summarize into the project's Obsidian strategy namespace if used;
   - update the active strategy queue with only the current direction, next safe action, and blocked activation surfaces.
6. Keep global memory to a compact signpost only if future sessions need routing help.
7. End with the next safest implementation slice, usually schemas/packets/local fixtures before UI, payments, OAuth, publishing, or other production activation.

## Good output shape

A useful strategy artifact includes:

- reviewer identity / route / visible model / limitations;
- recommended long-term goal and one-sentence positioning;
- explicit non-goals;
- initial wedge / delayed risky categories;
- north-star and supporting metrics;
- phase roadmap with exit criteria;
- subscription tier assumptions;
- unit-economics and break-even assumptions;
- 30/60/90-day validation cycle;
- next safe engineering slice;
- blocked actions requiring human approval.

## Pitfalls

- Do not present invented pricing, churn, CAC, or growth curves as facts. Label them as assumptions.
- Do not treat a strategic recommendation as approval to connect accounts, spend money, publish, schedule, charge customers, or ingest private customer data.
- Do not let the business-cycle artifact become the only active navigation layer; update the active queue so future workers see the current safe slice quickly.
- Do not store the full roadmap or financial model in Hermes global memory.
- Do not skip Obsidian/project-index updates when the project explicitly uses Obsidian for long-term strategy.
